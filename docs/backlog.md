# Product Backlog

## Metode Prioritisasi: MoSCoW
Prioritasi menggunakan metode MoSCoW untuk menentukan fitur yang masuk ke dalam MVP (Minimum Viable Product)[cite: 31].

| Label | Definisi |
|---|---|
| **Must-have** | Wajib ada di MVP. Tanpa fitur ini sistem tidak dapat berfungsi[cite: 31, 291]. |
| **Should-have** | Penting dan bernilai tinggi, tetapi bisa ditunda jika waktu tidak mencukupi[cite: 31, 293]. |
| **Could-have** | Nice to have — meningkatkan pengalaman pengguna tetapi bukan prioritas utama[cite: 31, 295]. |
| **Won't-have** | Tidak dikerjakan semester ini. Dicatat agar ekspektasi tetap realistis[cite: 31, 297]. |

## Must-have (MVP)
*Minimal 3 story wajib ada di bagian ini untuk memenuhi syarat praktikum[cite: 55, 291].*

| ID | User Story | Role | Referensi US |
|---|---|---|---|
| BL-01 | Sebagai tamu, saya ingin mendaftar dan masuk ke akun, agar saya dapat mengakses fitur AI dan menyimpan resep pribadi[cite: 120, 252]. | Guest | US-08 |
| BL-02 | Sebagai pengguna, saya ingin **menginput bahan makanan ke AI Gemini**, agar saya mendapatkan saran resep otomatis sesuai bahan yang tersedia[cite: 120]. | User | US-02 |
| BL-03 | Sebagai pengguna, saya ingin membuat dan mempublikasikan resep masakan saya sendiri secara manual, agar dapat dibagikan kepada pengguna lain. | User | US-03 |
| BL-04 | Sebagai pengguna, saya ingin melihat detail resep lengkap (bahan dan langkah), agar saya dapat mengikuti panduan memasak dengan benar[cite: 120]. | User | US-01 |
| BL-05 | Sebagai admin, saya ingin memoderasi resep yang dikirimkan, agar konten platform sesuai dengan standar kualitas[cite: 250]. | Admin | US-07 |
